You are a platform engineer synchronizing branches and resolving merge conflicts using documented strategies.

## Context

The GoldenPath IDP uses a `development → main` branching model (ADR-0042). Feature branches merge to `development` via PR, and `development` merges to `main` for releases. Syncing branches (especially after releases or when feature branches fall behind) frequently causes merge conflicts in predictable files: append-only logs, auto-generated indexes, and shared config files.

This prompt encodes the correct merge strategy for each file category so conflicts are resolved correctly and quickly.

## Your Task

Synchronize branches and resolve all merge conflicts using the documented strategy per file type.

## Preconditions

- [ ] Working tree is clean (`git status` shows no uncommitted changes).
- [ ] Both source and target branches are up to date with remote (`git fetch origin`).
- [ ] You know the sync direction (see Sync Direction Rules below).

## Sync Direction Rules

| Scenario | Command | When |
|---|---|---|
| After release: main → development | Merge `main` into `development` | After PRs merged to main |
| Feature branch behind development | Merge `development` into feature branch | Before PR creation |
| Stale sync PR | Merge target branch into sync branch | When sync PR shows conflicts |

**Protected branch note:** `development` is protected — you cannot push directly. If syncing main → development, create a `sync/main-to-development-{date}` branch and PR it to development.

## Conflict Resolution Strategy by File Type

### Append-Only Files (take target/newer version)

These files grow by appending entries. The target branch (usually `development`) has the most recent appends.

| File | Strategy | Reason |
|---|---|---|
| `session_summary/agent_session_summary.md` | Take theirs (target branch) | Target has newer session entries |
| `session_capture/*.md` | Take theirs (target branch) | Append-only, never edit prior entries |
| `.goldenpath/value_ledger.json` | Take theirs (target branch) | Running totals, target is more current |

```bash
git checkout --theirs session_summary/agent_session_summary.md
git checkout --theirs .goldenpath/value_ledger.json
git add session_summary/ .goldenpath/
```

### Auto-Generated Index Files (regenerate after merge)

These files are generated by pre-commit hooks. Don't manually resolve — accept either version and let the hook regenerate.

| File | Strategy | Regeneration |
|---|---|---|
| `docs/adrs/01_adr_index.md` | Accept theirs, then regenerate | `pre-commit run generate-adr-index --all-files` |
| `docs/50-scripts/01_script_index.md` | Accept theirs, then regenerate | `pre-commit run generate-script-index --all-files` |
| `reports/index_metadata.json` | Accept theirs, then regenerate | `python3 scripts/standardize_metadata.py docs/` |

```bash
git checkout --theirs docs/adrs/01_adr_index.md
git checkout --theirs docs/50-scripts/01_script_index.md
git add docs/adrs/01_adr_index.md docs/50-scripts/01_script_index.md
```

### Config Files (manual merge — keep both additions)

These files may have legitimate additions from both branches. Merge manually.

| File | Strategy |
|---|---|
| `CLAUDE.md` | Keep both branch additions, deduplicate |
| `Makefile` | Keep both targets (non-overlapping) |
| `.pre-commit-config.yaml` | Keep both hooks, check for version conflicts |
| `.github/workflows/*.yml` | Keep target version unless source has critical fixes |

### Governance Documents (prefer target branch)

ADRs, GOVs, PRDs, and changelogs that conflict should generally take the target branch version (it's more current).

| File Pattern | Strategy |
|---|---|
| `docs/adrs/ADR-*.md` | Take theirs (target), then verify frontmatter |
| `docs/governance/policies/GOV-*.md` | Take theirs (target) |
| `docs/changelog/entries/CL-*.md` | Take theirs (target) |

## Step-by-Step Implementation

### Phase 1: Assess

1. Fetch latest from remote:

   ```bash
   git fetch origin
   ```

2. Check current branch:

   ```bash
   git branch --show-current
   ```

3. Identify the merge direction and check for conflicts:

   ```bash
   # Preview conflicts without merging
   git merge --no-commit --no-ff origin/{source-branch} || true
   git diff --name-only --diff-filter=U
   git merge --abort
   ```

### Phase 2: Merge and Resolve

1. Start the merge:

   ```bash
   git merge origin/{source-branch}
   ```

2. If conflicts occur, categorize each conflicting file using the strategy tables above.

3. For append-only files:

   ```bash
   git checkout --theirs {file-path}
   git add {file-path}
   ```

4. For auto-generated files:

   ```bash
   git checkout --theirs {file-path}
   git add {file-path}
   ```

5. For config files, open and manually merge, keeping additions from both branches.

6. Complete the merge:

   ```bash
   git add .
   git commit  # Accept the default merge commit message
   ```

### Phase 3: Post-Merge Validation

1. Regenerate auto-generated indexes:

   ```bash
   pre-commit run --all-files
   ```

2. If pre-commit modified files, stage and commit:

   ```bash
   git add -A
   git commit -m "fix: regenerate indexes after merge"
   ```

3. Validate Terraform (if infra files changed):

   ```bash
   for env in dev test staging prod; do
     terraform -chdir=envs/$env init -backend=false && terraform -chdir=envs/$env validate
   done
   ```

4. Push:

   ```bash
   git push
   ```

### Phase 4: PR (if targeting protected branch)

If the target is `development` (protected), create a PR per PROMPT-0006:

```bash
gh pr create --base development --title "sync: merge main to development" --body "..."
```

Use Change Type: `Infra change` for sync PRs.

## Verification Checklist

- [ ] All conflicts resolved (no unmerged files in `git status`).
- [ ] Append-only files have the target branch version (no lost entries).
- [ ] Auto-generated indexes regenerated via pre-commit.
- [ ] `pre-commit run --all-files` passes clean.
- [ ] Config files have additions from both branches (nothing dropped).
- [ ] Branch is pushed to remote.

## Integration Verification

- [ ] If PR created: `pr-guardrails` check passes.
- [ ] If Terraform files changed: `terraform validate` passes for all environments.
- [ ] ADR index is current and complete.

## Do NOT

- Force-push after merge (`git push --force`).
- Take the source branch version for append-only files (you'll lose recent entries).
- Manually edit auto-generated index files (they'll be overwritten by hooks).
- Push directly to `development` (it's protected — use a PR).
- Delete conflict markers and guess at the resolution — use the strategy table.
- Use `git rebase` for branch sync (merge preserves history).

## Output Expected

1. List of conflicting files and resolution strategy used for each.
2. Post-merge `pre-commit run --all-files` output (clean).
3. PR URL if targeting a protected branch.
4. Count of files synced.

## Rollback Plan

- If merge not yet pushed: `git merge --abort` or `git reset --hard HEAD~1`.
- If merge pushed to a feature branch: `git revert -m 1 <merge-commit>`.
- If PR created: close PR and delete the sync branch.
